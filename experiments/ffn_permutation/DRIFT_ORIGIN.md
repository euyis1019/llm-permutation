# BF16 相对漂移的来源：浮点加法不满足结合律

> 本文单独回答一个问题：**正确的联动 permutation 数学上不改变函数，那 logits 上
> ~2% 的相对漂移（rel_l2≈2e-2）到底从哪来？**
> 所有实验证据来自 [`RESULT.md`](RESULT.md)（实验一）与
> [`../ffn_halfswap/RESULT.md`](../ffn_halfswap/RESULT.md)（实验二）。

## 1. 一句话回答

漂移来自 **`down_proj` 矩阵乘法里求和顺序的改变**：置换重排了 down 的列（连同它的
输入 `h`），同一组乘积换了一个**分组顺序**去累加。浮点加法满足交换律但**不满足结
合律**，所以求和结果在最低有效位上不同——这是有限精度算术的固有性质，与置换公式
的正确性无关。整条 FFN 里其他所有步骤（gate、up、SiLU、逐元素乘）都被实验证明
**逐 bit 不变**。

## 2. 最小可复现的例子

BF16 只有 8 位有效尾数，在 256 附近的表示间隔是 2：

```python
import torch
a = torch.tensor(256., dtype=torch.bfloat16)
b = torch.tensor(1.,   dtype=torch.bfloat16)

(a + b) + b   # = 256.0   （256+1=257 舍回 256，再 +1 又舍回 256）
a + (b + b)   # = 258.0   （1+1=2 精确，256+2=258 精确）
```

同样三个数，只因求和顺序不同，结果差了 2。`down_proj` 对每个输出元素做的是
9728 项乘积的累加：

```text
y_i = Σ_{k=0}^{9727} W_d[i,k] · h[k]
```

置换后变成 `y_i = Σ_k W_d[i, perm[k]] · h[perm[k]]`——**同一组乘积**，但 GPU 的
GEMM kernel 按固定的 tile/块结构分组累加，元素换了位置就落进了不同的分组，中间部
分和的舍入路径随之改变。于是 `y'` 与 `y` 在最低位上出现 ~1e-3 量级的相对差异。

## 3. 为什么其他步骤不产生漂移

对 gate/up 做**行置换**，每个输出元素 `g'_i = W_g[perm[i]] · x` 仍是原来那一行与
x 的点积——**点积内部的求和顺序完全没变**，只是结果换了个存放位置；SiLU 和
`g ⊙ u` 都是逐元素运算，跟着位置走。所以直到 down 的输入为止，一个浮点误差都
没有。

## 4. 实验证据链（每一条都是逐 bit 级）

| # | 实验 | 结果 | 说明 |
|---|---|---|---|
| 1 | `gate/up/乘积` 坐标对齐 | 与 baseline 重排后 **bitwise 相等**（120/120） | down 之前零误差 |
| 2 | canonical-down：把置换后的 `h'` 逆排回原序，送入**原** down（求和顺序不变） | 与 baseline **bitwise 相等**（120/120） | 排除"置换公式本身有误差" |
| 3 | native-down：`down[:,perm] @ h'`（求和分组改变） | rel_l2 ≈ 4.7e-3 | 漂移在此、且仅在此产生 |
| 4 | adjacent_swap 置换（只交换相邻两项：用到交换律，**分组不变**） | BF16 输出 **bitwise 不变**（15/15） | 精确锁定"结合律"是元凶——交换律精确成立 |
| 5 | reverse / random / half-swap（分组改变） | 均出现 ~5e-3 漂移 | 只要分组变就漂移，与置换"形状"无关 |
| 6 | 精度缩放：同一置换在 fp64 下误差 ~1e-14 | 误差随精度提高消失 | 舍入噪声的指纹；真正的公式错误是 O(1) 且不随精度变（fp64 负对照 rel_l2≈1 可证） |

结论：**2 + 4 两条把漂移逐 bit 定位在"down GEMM 的求和结合顺序"这一次浮点运算
上，排除了其他一切来源。**

## 5. 从单层 5e-3 到 logits 2e-2：传播与饱和

- 单层 down 输出注入 ~4.7e-3 的相对扰动；
- 扰动沿残差流经过后续层被反复变换，到 logits 时放大到 ~2e-2 并**饱和**：
  只置换 layer 0 一层（2.1e-2）≈ 置换全部 36 层（2.2e-2），只置换最后一层
  layer 35 则只有 3.8e-3（几乎没经过放大）；
- 决定量级的是**扰动经过的网络深度**，不是被置换的层数——多层注入的噪声互不
  相关，不做线性累积。

## 6. 它为什么（几乎）无害

- logits cosine ≥ 0.9993；top-1 翻转仅 ~2%，且**全部**发生在 baseline 中
  top1-top2 margin < 0.13 的近并列 token 上（全体 token 的 margin 中位数是
  1.875）——即只有"本来就五五开"的决策会翻；
- 温度 0 生成 32 条中 27–29 条逐 token 一致，分叉均为同义改写；
- 类比：同一笔 9728 项的账用两种顺序各加一遍——总数在最低有效位上差一点，
  绝大多数决策不受影响，只有本来就打平的决策可能翻向另一边。

## 6.5 定量细化（实验三补充）

后续的量级画像实验（[`../activation_magnitudes/RESULT.md`](../activation_magnitudes/RESULT.md)）
把本文的机制进一步定量分解：真实 GPU kernel 用 fp32 累加器，"小项在累加过程中被
吃掉"的效应实测只有 ~2.5e-7；可见漂移的主通道是**输出写回 BF16 格点时的边界翻转**
——单次 GEMM 固有误差 2.33e-3 ≈ 量化下限 ulp/√12（2.26e-3），置换漂移
3.1–3.2e-3 ≈ √2 × 下限（3.19e-3），即置换等价于把输出量化噪声独立重抽一次。
本文的"刻度/格点"图景不变，舍入发生的位置从累加内部移到写回输出那一步。

## 7. 如果需要消除漂移

| 方法 | 效果 |
|---|---|
| 用 fp32/fp64 权重与计算做置换对比 | 漂移降到 1e-5 / 1e-14 量级（随 eps 缩放） |
| 对比时用 canonical 等价形式（先逆排再乘原矩阵） | BF16 下也逐 bit 一致（仅适用于验证，不适用于真部署置换后权重） |
| 接受漂移，改用 margin-aware / logits 级 / 任务级指标 | 推荐：漂移不改变函数，exact-match 的假阳性来自近并列 token |
