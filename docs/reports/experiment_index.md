## 1. 预备知识

### 1.1 研究动机

这个系列的最初目的服务于一个与剪枝(pruning)和模型合并(merging)相关的项目，其中涉及到了 Permutation 的操作，也因为 Git-Rebasin 与 Thinking Machines Lab 的 [https://thinkingmachines.ai/blog/defeating-nondeterminism-in-llm-inference/](https://thinkingmachines.ai/blog/defeating-nondeterminism-in-llm-inference/)， 对此比较感兴趣

### 1.2 模型与术语

- 模型:Qwen3-4B-Base(预训练基础模型,下称 Base)与 Qwen3-4B(指令微调版,下称 Instruct)。两者同架构:36 层 Transformer,`hidden_size = 2560`,FFN `intermediate_size = 9728`。
- 中间通道(intermediate channel):FFN 中间向量的 9728 个维度之一。第 i 个通道由 `gate_proj` 第 i 行、`up_proj` 第 i 行、`down_proj` 第 i 列组成,即文献中的 FFN 神经元。注意与 `hidden_size = 2560` 区分,后者是残差流(residual stream,层与层之间传递的主干表示)的维度,本系列的 permutation 不触碰它。
- BF16(bfloat16):16 位浮点格式,尾数只有 8 位,相邻可表示数的相对间距约 1/128 ≈ 0.8%。全部实验的权重和计算都用 BF16。
- ULP:某个数值附近相邻两个可表示浮点数的间距。输出差 1 ULP 就是最低有效位翻了一下。

### 1.3 联动置换(valid permutation)

Qwen3-4B 的 FFN 是 SwiGLU 结构:

```
y = W_down · ( SiLU(W_gate · x) ⊙ (W_up · x) )
```

取一个置换 π,同时把 `W_gate`、`W_up` 的行和 `W_down` 的列按同一个 π 重排。每个中间激活分量只是换了位置,与它配对的 `down_proj` 列跟着换到同一位置,所以精确实数计算下输出 y 严格不变。这叫联动置换。只置换其中一两个矩阵,或三个矩阵用不同的 π,都会破坏配对,把输出彻底摧毁(实验一的负对照证实了这一点)。

### 1.4 为什么数学不变、GPU 结果仍会变

`down_proj` 的每个输出元素是 9728 项乘积的和。GPU 不按固定顺序累加,而是分块求部分和再逐级合并。浮点加法不满足结合律,置换虽然不改变"加哪些数",却可能改变这些项进入哪个归约块、以什么顺序合并,于是有限精度结果的最低几位可能不同。这个微小差异会随后续 35 层前向传播,并在两个候选 token 分数接近时改变贪心解码的选择。

### 1.5 常用指标

- `rel_l2`:两条张量之差的 L2 范数除以基准张量的 L2 范数。`rel_l2 = 0.01` 表示整体距离约为基准范数的 1%,不表示每个元素变了 1%。
- 逐比特(bitwise):原始字节完全相同,最强的等价。
- top-1 翻转:logits 最大值对应的 token 变了,贪心解码在这一步会选出不同的 token。
- margin:top-1 与 top-2 的 logits 之差。margin 越小,越容易被微小数值差异翻转。
- KL 散度:两个 softmax 后分布之间的距离
- eval-macro:六项 benchmark 准确率的等权平均。

### 1.6 两条推理栈(读数字前必须知道)

系列里有两套执行环境,logits 漂移的绝对数值不同,不能直接互比:


| 栈                                 | 用在哪些实验      | 全 36 层随机置换的 logits rel_l2        |
| --------------------------------- | ----------- | -------------------------------- |
| HF transformers + PyTorch,逐 token | 实验一、二、三     | 约 2e-2(全 token 中位数)              |
| vLLM,冻结确定性配置                      | 实验四、五(部分)、六 | 约 1e-2(最后一个 token,32 prompt 中位数) |


差异来自两点:kernel 不同(vLLM batch-invariant 的归约结构与 torch 不同,单层天花板低约 17 倍),测量点不同(全部 token 取中位 与 只看最后一个 token)。同一实验内部的对比总是同栈同口径的,跨实验引用时注意栈别。

vLLM 冻结配置指:`VLLM_BATCH_INVARIANT=1`、`enforce_eager=True`、关闭 prefix caching、温度 0、固定 GPU。这是因为默认配置的 vLLM 贪心解码不是逐次运行可复现的(实验四实测同权重重跑 GSM8K 有约 15/100 的答案翻转);冻结后 Base 达到完全确定,Instruct 只剩 GSM8K 一项残留波动。

### 1.7 工作方式

多数实验采用预注册:先写死实验计划(判据阈值、seed、停机规则),由另一个执行 agent(codex)照做,不允许中途改判据;硬判据失败就停机写失败报告,由 reviewer 复审。这个纪律在系列中两次发挥作用(实验五两轮都合规停机,复审发现是判据设计错误而非数据问题)。角色分工:用户定方向,reviewer(本文作者)设计与复审,codex 执行 GPU 实验。

## 3. 实验一:ffn_permutation(联动置换的正确性与误差定位)

问题:联动置换在数学和实现层面是否成立?BF16 下的残余误差出现在哪里、多大、与什么有关?

做法(HF transformers 栈,RTX 4090,BF16,2026-07-10):

- A:d=7、m=11 的合成矩阵做方向性单测和 fp64 校验;
- B:真实的第 0/17/35 层 MLP,5 种输入 × 8 种置换 × 10 组对照,共 159 个 case;
- C:完整模型,32 条固定 prompt × 23 个 case,外加 64-token 贪心生成对比。

关键结果:


| 观测                                        | 数值                                                                      |
| ----------------------------------------- | ----------------------------------------------------------------------- |
| fp64 下联动置换的误差                             | 约 1e-14,机器精度                                                            |
| 单 MLP,正确置换的 rel_l2(中位)                    | 4.72e-3                                                                 |
| 单 MLP,任何错误置换(单矩阵、错配、方向反)的 rel_l2          | 1.4–1.5,输出等同随机向量                                                        |
| 正确与错误的分离度                                 | 317 倍(最坏 235 倍),预注册要求 100 倍                                             |
| canonical-down 隔离(把置换激活恢复原序再进原 down_proj) | 120/120 逐比特等于 baseline                                                  |
| 全模型 all-36 置换的 logits rel_l2(全 token 中位)  | 2.0–2.2e-2                                                              |
| 全 token top-1 一致率                         | 97.8–98.5%,翻转全部集中在 margin 中位仅 0.03–0.125 的近并列 token(全体 margin 中位 1.875) |
| 64-token 贪心生成逐字一致                         | 26–29/32,分叉均为同义改写                                                       |
| 只置换 L35 / L17 / L0 的 logits rel_l2        | 3.8e-3 / 1.7e-2 / 2.1e-2                                                |


结论与保留:

1. **联动置换的对称性成立,方向约定已验证,可以作为后续匹配搜索的重参数化基元。** 合成 fp64 对照的误差约为机器精度,而真实 BF16 MLP 仍保留可测的低位漂移;这一区分同时验证了代数实现和有限精度边界。
2. 误差被三条独立证据(canonical-down 隔离、相邻交换逐比特不变、全模型 first-diff 定位)锁定为 `down_proj` GEMM 的归约顺序。联动置换只改变中间通道的排列;在 `gate_proj`、`up_proj`、激活函数和逐元素乘中,各通道仍可一一对应。到了 `down_proj`,这些通道要并行求和,换序会让乘积项进入不同的归约分组,因此首次产生舍入差异。
3. **影响大小由扰动传播深度主导:只置换第 0 层就接近全 36 层的效应,只置换最后一层几乎没有影响。**
4. 保留:虽然逐 token 的 top-1 翻转率约为 2%,一次翻转就可能让后续生成走向另一条路径。在 32 条固定 prompt 上,两族各有约 3 至 6 条生成文本不再逐字一致。因此,若把“生成文本必须完全相同”当作 merge 是否破坏模型的唯一判据,会把这类有限精度分叉误判成语义退化;更稳妥的做法是同时检查 margin、任务得分和人工可接受性。

产物:`experiments/ffn_permutation/RESULT.md`,原始数据在其 `results/`。Base 模型的独立结果记录在各实验目录的 `BASE_RESULT.md`。

## 4. 实验二:ffn_halfswap(平均绝对位移最大的规则置换)

问题:前一半通道与后一半通道整体互换(每层 `perm = [4864..9727, 0..4863]`)是平均绝对位移最大的规则置换,它有没有超出随机置换的特殊影响?温度 0 下重复运行有没有随机性?

*这里的位移最大按所有通道的平均绝对位移计算。前后半区互换后,每个通道都移动半个隐藏维度,因此平均值达到最大。它不表示某一个通道能移动到最远位置;整体反转也能达到相同的最大平均绝对位移,只是各通道的移动距离不均匀。*

做法:全 36 层 half-swap,32 条 prompt,每条件 forward 重复 2 次、贪心生成独立重复 8 次,先验证条件内确定性,再对比条件间差异。

关键结果:


| 观测                           | Instruct         | Base             |
| ---------------------------- | ---------------- | ---------------- |
| 条件内重复(128 次 forward、256 次生成) | 全部逐比特/逐 token 一致 | 全部逐比特/逐 token 一致 |
| logits rel_l2(全 token 中位)    | 2.13e-2          | 1.57e-2          |
| 全 token top-1 一致率            | 98.25%           | 98.01%           |
| 64-token 生成逐字一致              | 28/32            | 23/32            |


结论:**half-swap 的漂移落在随机置换的同一观测量级,置换的规则性和平均位移本身没有表现出额外影响。** 这推翻了“通道搬得越远,误差就应越大”的直觉,并把后续问题转向 kernel 实际如何分组归约。温度 0 的 HF 栈在本实验重复条件下是严格确定的,因此条件间差异可以归因于置换;实验五随后进一步确认,决定零漂移边界的是对齐归约块,不是几何位移大小。

产物:`experiments/ffn_halfswap/RESULT.md`、`BASE_RESULT.md`。

## 5. 补充实验，实验三:activation_magnitudes(数值量级画像与天花板闭环)

问题:模型内部各处的数值有多大?BF16 在这些量级下的分辨率是多少?置换漂移的大小有没有一个第一性的解释?

做法:9 个 benchmark 各采 16 条 prompt(共 144 条、16132 个 token),测量 embed、残差流、FFN 各中间量、logits 的量级分布;对 L0/17/35 的 `down_proj` 做逐项分解(对消比、bf16/fp32/fp64 三层误差分解)。模型只读。

关键结果:

1. 内部数值横跨 6 个数量级:embedding 约 0.02,残差流 RMS 约 32,个别通道达到 16384,Instruct logits 约 4。BF16 的相对分辨率近似固定,所以数值越大的位置,相邻可表示值之间的绝对间隔也越大。这意味着同样是最低位跳动,在大幅值中间量上对应更大的绝对变化;这些变化进入后续对消和层间传播后,可能留下可测的输出差异。
2. `down_proj` 求和存在 30 到 70 倍的正负对消。可以把它理解为先累加许多幅值较大的正负项,最后只留下较小的净结果。不同求和顺序会在中间步骤产生不同舍入;当净结果来自强烈对消时,这些中间舍入相对最终剩余量就更显眼。
3. 单次 BF16 down-GEMM 的固有误差约为千分之 2.32,与 BF16 输出舍入的理论水平千分之 2.26 几乎相同。置换后两种规约顺序的输出差约为千分之 3.09 到 3.22,也接近两份独立舍入结果之间的预期差异千分之 3.19。这说明置换主要换了一条舍入路径,没有引入新的大误差来源。
4. Base 的 logits RMS 为 8.67,Instruct 为 3.77,两者的输出尺度明显不同,但 down-GEMM 的固有相对误差都约为千分之 2.2 到 2.3。这说明误差主要来自 BF16 矩阵规约,不依赖模型是否经过指令训练。

结论:置换改变了 BF16 的规约和舍入路径,两种顺序之间的差异接近两份独立舍入结果的差异。在当前三层与当前 kernel 的测量范围内,单层漂移受 BF16 舍入水平约束。**结合实验五对多种置换族的独立测量可以进一步说:一旦置换实质改变归约分组,继续增加通道位移通常不会让单层误差按比例增大。** 这是当前执行栈中的经验边界,不是对其他 kernel 的普适上界。

产物:`experiments/activation_magnitudes/RESULT.md`、`BASE_RESULT.md`。

## 6. 实验四:ffn_benchmark_eval(benchmark 层的影响)

**核心问题:logits 层约 1–2% 的漂移传到 benchmark 分数上是多大?与置换的层位置、位移方式有什么关系?**

做法(vLLM 冻结配置,76 个 checkpoint,每个跑 6 项 benchmark 并保存逐题记录):

- 噪声地板:每族 11 个同权重 baseline(原始 ×2、重存副本、8 个重复),两两配对得到"同函数重跑"的波动;
- 主实验:每族 20 个 all-36 随机置换 seed(s1000–1019);
- 消融(仅 Instruct):层范围轴(single L0/L17/L35、prefix6/18、all36)和位移轴(相邻交换、整体倒序、随机)。

样本:MMLU/C-Eval/CMMLU/GSM8K 各 500 条,HumanEval+ 164、MBPP+ 378 全量。

关键结果:

1. 确定性:Base 的 11 个 baseline 在全部 6 项 benchmark 上逐题 correctness 完全一致(地板为 0);Instruct 除 GSM8K 外也全部一致,GSM8K 残留同权重波动最大 2.40 pp、逐题不一致率 5.97%。
2. 20 组随机置换的结果分布(六任务平均分相对各自 baseline):Instruct −0.79 ± 0.19 pp,20/20 为负;Base +0.49 ± 0.21 pp,20/20 为正。每组都由不同随机种子生成一套全 36 层置换。每层对 9728 个通道做不重复、不遗漏的一一重排,不同层各自生成顺序,同层三个 FFN 矩阵正确联动,其他权重不变。
3. 逐 benchmark:三个多选任务在两族的 20 组随机置换上全部落在 ±1 pp 内,逐题一致率 ≥99%;偏移集中在 GSM8K(Instruct −2.33)、HumanEval+(Instruct −1.71)、MBPP+(Base +1.03)、GSM8K(Base +0.97)。
4. 消融:single L0 已产生 all-36 约 75% 的影响,single L35 几乎为零;层位置远比层数量重要。
5. 勘误(07-12 追加):相邻交换臂原报告"影响约为随机的 1/4",复审后更正为与零效应完全一致(6 项中 5 项不一致率恰为 0,唯一非零的 GSM8K 完全落在 Instruct 自身噪声地板内)。原因见实验五:该相邻交换是偶对齐的,属于免费类。

结论与保留:影响小但真实(Base 地板为零,偏移不可能是运行噪声);集中在长生成任务;方向随模型固定但两族相反,当时未能解释方向来源(见实验七)。Instruct 的 GSM8K 数字始终要对照其 2.4 pp 的自身波动解读。

产物:`experiments/ffn_benchmark_eval/RESULT.md`(勘误在其 §8),逐题原始记录在 `results/raw/<tag>/<bench>.raw.json`。

## 7. 实验五:permutation_min_cost

问题:如果必须置换一部分通道,怎么组织置换才能让数值代价最小?存不存在规律?

### 7.1 第一轮:连续定律假设失败,复审改写为三档定律

做法:8 个置换族 × 5 seed ≈ 147 个实例 × 3 层 × 3 种输入 = 1323 条单层 `down_proj` 测量,同时记录 6 个几何量(位移、跨块数、逆序数等)做回归。

结果:预注册判据 S1a(某几何量 R² ≥ 0.8 或 ρ ≥ 0.9)与 S1b 失败,合规停机。复审发现数据的真实形状是三档阶跃而非连续曲线,连续回归的判据形式本身错了:


| 档位  | 条件                                 | torch bf16 单层漂移     |
| --- | ---------------------------------- | ------------------- |
| 免费  | 每个通道留在自己的对齐 8 块内(⌊i/8⌋ = ⌊π(i)/8⌋) | 恰好 0,逐比特            |
| 中间  | 只在对齐 16–32 窗内混合                    | 约 3e-5 至 1e-4       |
| 饱和  | 实质跨块                               | 2.4e-3 至 3.2e-3,天花板 |


三个要点:判据是对齐而不是位移(偶对齐相邻交换逐比特免费,奇对齐同样位移为 1 却直达天花板);饱和是开关不是斜坡(散置 5% 与 50% 的通道漂移一样,都在天花板);过渡区无法用先验指标精确预测(激活能量假说 R² 仅 0.03–0.30),不能当设计旋钮。

### 7.2 第二轮(v1.1):免费类正式确认,发现 torch GEMV 例外

做法:全新 seed 301–305,1248 条测量,免费类在 4 个 backend × 形状组合上做逐比特硬判据。

结果:硬判据 S1b-1 失败,合规停机;复审裁定为适用域画错而非定律证伪:


| 组合                           | 免费类逐比特             |
| ---------------------------- | ------------------ |
| torch bf16 GEMM(输入行数 ≥ 2)    | 126/126 通过         |
| vLLM batch-invariant prefill | 126/126 通过         |
| vLLM batch-invariant decode  | 126/126 通过         |
| torch bf16 GEMV(输入 1 行)      | 53/126,73 条失败全在此路径 |


GEMV 的归约结构没有连续 8 块,任何置换(含块内)都有 ≤6e-5 的偶发微噪声(每次只有 2560 个输出中的 1–3 个变),同时也没有天花板。另外 vLLM 的"饱和档"实为 2e-5 至 2e-4 的弥散小带,不像 torch GEMM 那样有紧天花板。

结论:部署栈(vLLM)上免费类完整成立;块宽 8 是当前 GPU、kernel、dtype 的实测属性,换栈必须重测。"要置换 30% 通道怎么最省"的答案:能设计成块内重排就严格零代价;必须跨块则跨多少不影响代价,可控的杠杆只剩动哪些层。

产物:`experiments/permutation_min_cost/OVERVIEW.md`(完整讲解),`review/REVIEW_stage1.md`、`review/REVIEW_stage1b.md`(两轮复审),两份 FAILURE_REPORT,原始测量 `results/stage1*.jsonl`。

## 8. 实验六:noise_floor(最终预注册轮:端到端验证 + 高斯扰动标定)

问题(四个):(0) 冻结配置下重复运行在 logits 层逐比特确定吗?(1) 块内置换在完整模型端到端是否零差,包括 benchmark?(2) 32-prompt logits 漂移能否预测 benchmark 不一致率?(6) 多大的高斯权重扰动才会产生超过置换的 logits 漂移?

做法(codex 按预注册计划执行,Base 模型为主,vLLM 冻结配置,全程 GPU 0):

- Part 0:同一 identity 权重,两个独立进程各跑 32 prompt,逐字节比较最后 token logits;
- Part 1a:免费类(F9 块内重排)、奇对齐相邻交换(F10)、全局随机(F7)的 32-prompt logits;1b:三个置换 checkpoint 跑全部 6 项 benchmark 与 baseline 逐题比对;
- Part 2:实验四的 8 个 Instruct 消融锚点,补测 32-prompt logits 漂移,与已有 benchmark 不一致率做 Spearman 相关;
- Part 6:RandOpt 式全模型 BF16 原位高斯噪声,σ ∈ {1e-6 … 1e-2} 10 档 × 3 seed,每个测 32-prompt logits;另做 CPU 端权重级量化统计。

关键结果:


| 判据            | 结果  | 内容                                                                         |
| ------------- | --- | -------------------------------------------------------------------------- |
| S0-1(硬)       | 通过  | 重复运行 32/32 逐比特一致,重复运行基线成立                                                  |
| S1-1(硬)       | 通过  | 块内置换与 identity 32/32 逐比特一致                                                 |
| S1-3(硬)       | 通过  | 块内置换 6 项 benchmark 逐题 correctness 零差(GSM8K 497/500 响应文本逐字节一致,3 条文本不同但对错不变) |
| S1-2/S1-4(软)  | 通过  | F10 与 F7 的 rel_l2 中位数 0.00949 / 0.00988,同带                                 |
| S2-1(软)       | 失败  | 锚点 logits 漂移与 benchmark 不一致率 Spearman ρ = 0.595,未达 0.9                     |
| P6-1/P6-2(预测) | 失败  | σ=1e-6 已 0/96 逐比特、漂移即达置换参考线的 1.0065 倍;σ* 只能记为 ≤1e-6                        |


高斯扰动曲线(权重量化吞噬 + logits 漂移):


| σ    | BF16 权重实际改变比例 | logits 中位 rel_l2 | 相对置换参考线 |
| ---- | ------------- | ---------------- | ------- |
| 1e-6 | 1.31%         | 0.00994          | 1.01 倍  |
| 1e-5 | 11.7%         | 0.00992          | 1.00 倍  |
| 1e-4 | 64.4%         | 0.0156           | 1.58 倍  |
| 1e-3 | 95.7%         | 0.1007           | 10.2 倍  |
| 1e-2 | 99.0%         | 1.0824           | 109.6 倍 |


事后补充(post-hoc,证据权重低于预注册):S1 把 σ 压到 1e-8(只改动约 53 万个参数,占 0.013%),漂移仍为参考线的 0.89–1.00 倍,已测范围内没有逐比特零区;S2 只扰动 FFN 三矩阵,σ=1e-6/1e-5 为参考线的 0.87–1.07 倍,排除"地板主要来自非 FFN 参数";S3 行为臂,σ=1e-4 的 5 个 seed GSM8K 全部落在 20 个置换 seed 的范围(76.2–79.2%)内,σ=1e-3 有 2/5 低于该范围。

Reviewer 纯分析部分(设计先于执行,用落盘数据计算):

- Part 3(选择偏差):在一半题目上从 20 个置换模型选最优,换另一半题目后约 0.6 pp 的选择优势基本消失(Base +0.631 → +0.070,Instruct +0.617 → −0.058)。功能相同的模型之间做选择,选出的优势不迁移。
- Part 4(baseline 排名):Base baseline 低于全部 20 组随机置换,Instruct 高于全部 20 组;只完成分布与排名,未完成逐题分解(实验七补上了一部分)。
- Part 5(答案投票):20 个置换模型对 GSM8K 做多数投票,Base 79.4%(baseline 77.2%,最佳单 seed 79.2%),Instruct 76.8%(75.6% / 74.6%)。仅此一个 benchmark、无显著性检验,是待复现的观察。
- Part 7(零阶方法信噪比):原设计的 NLL 比较存在统计口径不匹配(逐 prompt 绝对值均值 对 先平均再取绝对值),结果不能用作 MeZO/FZOO 的信噪比结论,记为未完成。

结论与保留:块内置换的零代价从单层 kernel 一路验证到 benchmark 逐题层面,这条链闭合了。高斯扰动在 σ 低至 1e-8 时仍实际改变约 53 万个 BF16 参数,其 logits 漂移与全层置换落在相近尺度;这说明当前模型会放大这种稀疏但仍覆盖大量参数的微小扰动,不能外推为单个参数或任何微小改动都会触发同一参考带。置换不改变精确函数而高斯真的改变函数,两者相近也不能直接推出"存在通用的 1% 测量下限"。32-prompt 最后 token 探针未能达到预注册的 benchmark 不一致率相关性判据(S2-1 失败),原因见实验七。

产物:`experiments/noise_floor/EXPERIMENT_PLAN.md`(冻结设计)、`EXECUTION_REPORT.md`、`SUPPLEMENT.md`、`results/acceptance_noise_floor.json`(机器验收,保留全部失败记录)、`reviewer_analysis/part3/4/5/7*.json`。

## 9. 实验七:两个 reviewer 补充分析(2026-07-13)

零 GPU,全部从已落盘数据计算。脚本在 `experiments/noise_floor/scripts/`,结果在 `reviewer_analysis/`。

### 9.1 行为层指标(metrics_supplement)

问题:rel_l2 只量整体距离,不回答"贪心解码的行为变没变"。用落盘的 fp32 logits 补算 top-1 翻转、margin、softmax KL、top-10 重叠,共 60 个单元 × 32 prompt,并与运行时记录的 rel_l2 逐条交叉校验(0 条不符)。

关键结果:


| 条件                                    | top-1 翻转(/32,3 seed 合并为 /96) | KL 中位(nats)  | top-10 重叠中位 |
| ------------------------------------- | ---------------------------- | ------------ | ----------- |
| 块内置换 F9                               | 0                            | 0            | 1.0         |
| 全局随机置换 F7                             | 2/32                         | 1.2e-3       | 1.0         |
| 高斯 σ ≤ 3e-5(含 post-hoc 的 1e-8 至 3e-7) | 5–7/96                       | 约 1.0–1.4e-3 | 1.0         |
| 高斯 σ = 1e-4                           | 9/96                         | 约 2.8e-3     | 0.9         |
| 高斯 σ = 5e-4                           | 18/96                        | 约 3.9e-2     | 0.8         |
| 高斯 σ = 1e-3                           | 42/96                        | 约 1.2e-1     | 0.6–0.7     |
| 高斯 σ = 1e-2                           | 96/96                        | 约 9          | 0.0         |


另外三个观察:

1. 翻转发生在低 margin 处:被翻转 prompt 的 baseline margin 中位 0.25,未翻转的 0.5625,与实验一的 margin 机制一致。
2. 在 σ ≤ 3e-5 的整个区间,四个描述性行为指标与置换参考条件处于相近范围,没有显示明显分离;本分析没有做统计等价性检验。行为层的明显分离从 σ = 5e-4 起出现。rel_l2 在 1e-4 已升到 1.58 倍,说明 rel_l2 比这些行为指标更早察觉扰动,两类指标互补。
3. Instruct 的 8 个消融锚点在 32-prompt 探针上几乎无信号(KL 中位约 1e-7,翻转 0–1 条),但这些 checkpoint 在 benchmark 上有 −0.2 至 −0.85 pp 的实测差异。这直接解释了 S2-1 相关性失败:探针本身对这批锚点太弱,不是 benchmark 数据有问题。附带发现:同一批 32 条 prompt 上,Instruct 的 baseline margin 中位 9.5,Base 只有 0.53,与实验三"Instruct margin 更大"的方向一致,提示 Instruct 在探针 prompt 上远离决策边界,探针灵敏度天然更低。

### 9.2 方向按 benchmark 分解(direction_decomposition)

问题:Base 全正、Instruct 全负的六任务平均分偏移分布在哪些 benchmark?其中多少可能是 baseline 自己的运行运气?

关键结果(offset = 20 组随机置换的均值减 baseline,单位为 pp;sign = 高于或低于 baseline 的置换组数量):


| benchmark  | Base offset(sign) | Instruct offset(sign) |
| ---------- | ----------------- | --------------------- |
| MMLU       | +0.45(19+/0−)     | +0.05(9+/5−)          |
| GSM8K      | +0.97(19+/1−)     | −2.33(0+/20−)         |
| C-Eval     | +0.45(17+/0−)     | +0.20(14+/1−)         |
| CMMLU      | +0.18(13+/2−)     | −0.57(0+/19−)         |
| HumanEval+ | −0.12(5+/9−)      | −1.71(1+/19−)         |
| MBPP+      | +1.03(19+/1−)     | −0.37(5+/14−)         |


三个发现:

1. Instruct 的负偏移集中在生成类:GSM8K 与 HumanEval+ 两项贡献了 −0.79 pp 中的 −0.67。Base 的正偏移则是弥散的,连三个多选任务都有 13–19/20 的一致正号,不限于生成类。
2. Instruct 的 GSM8K baseline 是一次幸运运行:baseline run1 得 75.6%,而它自己的 10 次同权重重跑分布在 73.2–75.6%(均值 74.58%),run1 恰好是最高值。若以重跑均值为参照,GSM8K 偏移从 −2.33 缩到 −1.31 pp,六任务平均分从 −0.79 缩到约 −0.62 pp。其余 benchmark(两族)重跑范围全部为 0,这一修正只影响 Instruct 的 GSM8K。
3. HumanEval+ 是最干净的真实布局效应:重跑范围为 0(无运行噪声可言),20 组随机置换中仍有 19 组低于 baseline,平均低 1.71 pp。逐题层面的系统性题目很少(每个 benchmark 只有 0 到 5 题被绝大多数置换组一致翻转),偏移主要由弥散的小幅翻转累积而成。

解读(写入报告时的口径):Instruct baseline 的高位置由两部分组成,约 0.17 pp 的六任务平均分来自 GSM8K 的单次幸运运行,剩余约 −0.62 pp 是真实的布局效应且集中在长生成任务,与"后训练 checkpoint 选择可能耦合了原始布局下的生成类跑分"的假说方向一致但不构成证明。Base 的原始布局则是全面地略低于其置换邻域(六项中五项一致为正),更接近"邻域里一个普通偏低的样本"。

产物:`scripts/metrics_supplement.py`、`scripts/direction_decomposition.py`,结果 `reviewer_analysis/metrics_supplement.json`、`reviewer_analysis/direction_decomposition.json`。

## 10. 结论清单(按证据强度)

预注册硬判据确认(最强):

1. 联动置换代数等价,BF16 误差唯一来源是 `down_proj` 归约顺序(实验一,判据 6/6)。
2. 保持对齐 8 块的置换在 torch GEMM(T≥2)、vLLM batch-invariant prefill、decode 上逐比特免费,378/378(实验五 v1.1)。
3. 冻结配置下重复运行逐比特确定;块内置换端到端 logits 逐比特、benchmark 逐题零差(实验六 S0-1/S1-1/S1-3)。

预注册数据得出(强):

1. torch T=1 GEMV 无 8 块结构,任何置换有 ≤6e-5 微噪声、无天花板(实验五 v1.1 的失败内容本身)。
2. 20 组随机置换的 benchmark 偏移:Instruct −0.79 ± 0.19 pp 全负,Base +0.49 ± 0.21 pp 全正;多选任务两族全部 ±1 pp 内(实验四)。
3. σ=1e-6 的高斯扰动(仅 1.31% 权重实际改变)logits 漂移已达全层置换参考线;σ* ≤ 1e-6(实验六 Part 6,预测失败即此发现)。

复审事后分析与对照充分的结论(中):

1. 饱和档高度 ≈ BF16 输出量化噪声的 √2 倍,是天花板不是累积量;跨块是开关不是斜坡(实验三闭环 + 实验五复审)。
2. 影响 ∝ 归约扰动 × 传播深度;single L0 ≈ all-36 的 75%,single L35 ≈ 0(实验一、四)。
3. 相邻交换勘误:偶对齐相邻交换与零效应完全一致,原"1/4 影响"是把噪声读成了效应(实验四 §8)。
4. 置换与低档高斯扰动在行为指标(翻转/KL/top-10)上处于相近观测范围,尚无统计等价性结论;翻转集中在低 margin prompt(实验七)。
5. Instruct 负偏移集中在生成类,其中约 0.17 pp 是 GSM8K baseline 的单次幸运运行;HumanEval+ 的 −1.71 pp 是无运行噪声混淆的真实布局效应(实验七)。
6. 功能相同模型之间的选择优势不迁移(约 0.6 pp 的 best-of-20 优势在换半数据后归零,实验六 Part 3)。

观察与待复现(弱):

1. GSM8K 上 20 模型投票高于 baseline 和最佳单 seed(两族都观察到,无显著性检验,实验六 Part 5)。
2. S3:σ=1e-4 的高斯扰动在 GSM8K 上落入置换分布的最小值到最大值范围(仅范围比较,post-hoc,不能视作等价性检验)。

## 11. 未决问题

1. 方向的机制:Instruct baseline 为什么在邻域高位(选择耦合假说未证明),Base 为什么弥散偏低。可做的检验:换一个同族但不同发布批次的 checkpoint 看方向是否保持;对生成类逐题做翻转重叠分析。
2. 零阶方法的信噪比:需要配对的 `θ+εu` 与 `θ−εu` 探针(同方向、同数据、统一 FP32 损失聚合),Part 7 的口径问题使现有数字不可用。
3. 高斯漂移地板的来源分解:σ=1e-8 仍达参考线,是深层放大还是量化与 kernel 路径的共同作用,现有实验分不开。
4. 原应用目标未动:两个真实 expert 的匹配与合并(对照组至少含 naive merge、随机匹配、权重匹配、激活辅助匹配),以及 MoE 场景(需同步处理 router 索引)。
5. 换栈重测:块宽 8 与各档数值全部是当前 RTX 4090 + PyTorch + vLLM 0.24 + BF16 的实测属性,任何硬件/kernel/dtype 变更后先跑逐比特 smoke test。
