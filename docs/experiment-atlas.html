05
Structured permutation

half-swap：换一种置换形状，现象仍一致

每层把 9728 维中间轴的前后 4864 维整块交换，36 层全部应用。这是大位移、自逆、结构化的置换。

与 baseline 文本

logits rel_l2
2.13e−2

与 all-36 随机置换的 2.0–2.2e−2 同量级。

它排除了“只有随机置换才这样”

half-swap、reverse 与 random 都会显著改变 down GEMM 的物理分组，因而表现出同量级 BF16 漂移。决定因素不是 permutation 看起来是否随机，而是它对底层布局改变了多少。

生成任务：小而真实，方向不普适

GSM8K / HumanEval+ / MBPP+ 可出现 1–2.3pp 级偏移；instruct 整体略降，Base 整体略升，说明不能把 permutation 说成普遍增益或普遍退化。

绝对 baseline 分数是多少？

Scope × magnitude

统一机制

当前证据支持两个关键因素：是否改变 down-proj 的物理归约分组，以及扰动还要向后传播多深。 这是由 logits 实验、half-swap、量化分解、复审勘误与 benchmark 消融共同支持的机制性总结，不是一个已拟合的连续预测规律。

08
Validity & caveats

为什么可信，以及哪里仍需克制

数值效应很小，因此“系统本身是否稳定”与“结论适用到哪”比多跑几个 seed 更重要。

baseline copy

重存 checkpoint 不引入偏移

不置换、按同样流程重存的 checkpoint 与原始 baseline tensor 逐字节一致，全部 benchmark 逐样本 Δ=0.00pp。

null distribution

11 个同权重 baseline / 模型

Base 六个任务 correctness 地板全部为 0；instruct 除 GSM8K 外也为 0，因此多数偏移不能归因于重跑噪声。

deterministic serving

冻结 vLLM 推理路径

VLLM_BATCH_INVARIANT=1、eager、关闭 prefix cache，并把每个模型族固定同一 GPU。

paired records

保存逐样本输出

不是只比较聚合 accuracy；456 份 raw 文件全部 complete，共 193,192 条 checkpoint×sample 记录。

最重要的三项限定
  • instruct GSM8K 有残余 null 噪声。 同权重两两重跑 max|Δ|=2.4pp、mean disagreement=5.97%，因此它的 −2.33pp 部分混淆；Base 六项 null 全为 0，HumanEval+ 等任务也提供独立真实效应证据。
  • suite 级严格预注册判据没有通过。 instruct 19/20、Base 11/20 seed 超出 ±0.5pp，所以不能写成“BF16 benchmark 完全等价”；准确表述是“小而可测，且任务依赖”。
  • 两套执行环境不同。 logits/activation 主实验为 PyTorch 2.13 + Transformers 4.57.6；benchmark 为 vLLM 0.24 + PyTorch 2.11 + Transformers 5.13。跨实验可谈机制一致性，不能假设 kernel 逐 bit 可比。
中途 GPU 配置变化是否污染结果？

benchmark 中途因资源占用把 gpu_memory_utilization 从 0.90 降到 0.28，并从双卡调度改为固定 GPU0；项目对 0.40 vs 0.90 在 Base GSM8K + MMLU 做了逐样本验证，correctness 无差异。该偏离已在原报告记录，但它仍提醒读者：结论绑定于当前硬件与确定性配置。

这些实验是否已经证明 permutation merge 有效？

没有。它们证明联合 permutation 是合法重参数化，并量化 BF16 风险；当前仓库尚未完成两个独立 checkpoint 的 assignment 搜索或 merge 后任务评估。
